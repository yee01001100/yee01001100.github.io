SQL注入是通过将恶意SQL代码插入Web表单、URL参数或查询字符串中，利用应用程序对用户输入验证不足的漏洞，欺骗数据库执行非授权操作的安全威胁，其核心在于**数据与指令的混淆**，而绕过技术则针对安全防护机制设计，旨在规避检测规则。

## 一、SQL注入原理详解

### 1. 根本成因
- **数据与指令混淆**：应用程序将用户输入直接拼接到SQL查询中，未区分数据与指令的语义。
- **过滤不严**：对特殊字符（如单引号、分号）和SQL关键字（如SELECT、UNION）未进行有效过滤或转义。
- **动态拼接**：使用字符串拼接方式构造SQL语句，而非参数化查询。

### 2. 工作机制
当应用程序执行类似以下代码时：
```sql
SELECT * FROM users WHERE username = '" + userInput + "'"
```
攻击者输入`admin' OR '1'='1`，将使查询变为：
```sql
SELECT * FROM users WHERE username = 'admin' OR '1'='1'
```
由于`'1'='1'`恒真，攻击者可绕过身份验证获取所有用户数据。

### 3. 注入类型
**1. 数字型注入**：参数无引号包裹（如`id=1`），直接拼接条件（`id=1 AND 1=2`）。

**核心逻辑：控制真假，观察反馈**

攻击者通过构造 `AND 1=1` 和 `AND 1=2`，目的是强行控制 SQL 语句最终的逻辑结果（True 或 False），然后观察网页的反应。

假设原本的 SQL 语句是：

```sql
SELECT * FROM users WHERE id = 1
```
第一步：测试 True 的情况
攻击输入： `id=1 AND 1=1`
生成 SQL： `SELECT * FROM users WHERE id = 1 AND 1=1`
逻辑结果： `True AND True` -> 结果还是 True
页面表现： 页面正常显示，数据加载正常，或者显示“用户存在”。
第二步：测试 False 的情况
攻击输入： `id=1 AND 1=2`
生成 SQL： `SELECT * FROM users WHERE id = 1 AND 1=2`
逻辑结果： True AND False -> 结果变成 False
页面表现： 页面可能显示空白、报错、显示“用户不存在”、或者内容缺失。

**输入的内容（AND 1=2）被数据库执行了，并且改变了查询的逻辑。**

这就确认了该参数存在 SQL 注入漏洞。接下来，攻击者就可以利用这种“真假反馈”的差异，一点点猜解出数据库里的数据（比如猜解表名、字段名）。

**2. 字符型注入**：参数被引号包裹（如`name='admin'`），需闭合引号（`name='admin'--`）。

在正常的程序逻辑里，单引号是用来把数据关在笼子里的。
比如程序代码是：`SELECT * FROM users WHERE name = ' + 用户输入 + '`
正常情况：你输入 admin。数据库看到的是 ... name = 'admin'。这里的 admin 被两个单引号夹在中间，数据库就认为它只是一个普通的名字（数据）。
攻击情况：攻击者输入 admin'（注意后面多了一个单引号）。
当这个输入拼接到SQL语句中时，变成了：... name = 'admin''。
关键点：攻击者输入的那个单引号，强行闭合了程序原本用来包裹数据的第一个单引号。
后面输入的内容就不再被视为“名字”这个数据，而是被视为SQL指令了。



**利用OR后的嵌套SELECT**

- **绕过登录验证**

经典的SQL注入攻击：

```sql
SELECT * FROM users WHERE username = 'admin' -- ' AND password = '123456'
```

通过`--`注释掉密码验证部分，直接以admin身份登录。

- **数据窃取**

黑客可以构造如下查询获取敏感数据：

```sql
SELECT * FROM users WHERE username = 'admin' OR 1=1; -- 获取所有用户数据
SELECT * FROM users WHERE username = 'admin' OR (SELECT COUNT(*) FROM admin) > 0; -- 检查admin表是否存在
```

通过这种方式，黑客可以逐步探测数据库结构并获取数据。

**3. 盲注**：无直接数据回显，通过布尔响应（真/假）或时间延迟（`SLEEP(3)`）推断数据。

**机制原理：**
当无论条件真假，页面返回的内容都一模一样（没有布尔差异）时，黑客就会利用**“时间差”**。
黑客会让数据库执行一个“延时指令”（如 `SLEEP(3)`），意思是“暂停3秒再回答”。

如果数据库执行了 `SLEEP`，页面会卡顿3秒才打开。

如果数据库没执行，页面瞬间打开。

---

黑客想知道管理员密码的第一位是不是 'a'。

**构造语句：**
`SELECT * FROM users WHERE id = 1 AND IF(SUBSTRING(password, 1, 1) = 'a', SLEEP(5), 0);`

- **逻辑翻译**：如果密码第一位是 'a'，就让数据库睡5秒；否则立刻返回。

**观察结果：**
黑客发送请求，盯着浏览器底部的加载条。

如果浏览器转了5圈才加载出来 -> **猜对了**。

如果秒开 -> **猜错了**。

- **报错注入**：利用数据库错误信息泄露数据（如`AND (SELECT 1 FROM (SELECT COUNT(*),CONCAT(0x3a,version(),0x3a) AS x FROM information_schema.tables GROUP BY x))`）。

## 二、常见绕过技术与原理

### 1. 编码绕过
- **URL编码**：将单引号`'`编码为`%27`，空格编码为`%20`，如`id=1%27%20AND%2011`。
- **双重URL编码**：对已编码字符再次编码（`%27`→`%2527`），需服务端二次解码才能生效。
- **Unicode编码**：用`u0027`替代单引号，部分WAF无法识别。
- **十六进制编码**：将字符串转换为十六进制（`'admin'`→`0x61646D696E`）。

### 2. 关键字混淆
- **大小写混合**：利用SQL不区分大小写特性，如`SeLeCt`、`UnIoN`。
- **双写关键字**：若过滤`SELECT`，可构造`SELSELECTECT`，过滤后剩余`SELECT`。
- **注释符拆分**：用`/**/`替代空格，如`UNI/**/ON SEL/**/ECT 1,2`。
- **内联注释**：MySQL支持`/*!50000UNION*/`语法，仅在特定版本执行。

### 3. 逻辑与函数替换
- **逻辑符号替换**：`AND`→`&&`，`OR`→`||`，如`id=1%27||11`。
- **函数等价替换**：`SLEEP(3)`→`BENCHMARK(1000000,1)`，`SUBSTRING`→`MID`。
- **无逗号技巧**：`SUBSTR(database() FROM 1 FOR 1)`替代`SUBSTR(database(),1,1)`。

### 4. 特殊场景绕过
- **宽字节注入**：GBK编码中`%df%27`组合为`運'`，使单引号逃逸（`id=1%df%27 AND 11`）。
- **二次注入**：先将恶意数据存入数据库（如`admin' #`），后续查询时触发注入。
- **空字节绕过**：在过滤器与应用处理差异处，用`%00`截断检测（`category=book%00';DROP TABLE test`）。

### 5. WAF对抗技巧
- **动态查询**：将SQL转为ASCII十六进制字符串执行（`DECLARE @q VARCHAR(200); SET @q=0x44526f50205461526c65; EXEC(@q)`）。
- **PHP正则回溯绕过**：构造超长字符串（`union/*100万个a*/select`）触发`pcre.backtrack_limit`限制。
- **参数污染**：`id=/*a=1*/1/*b=2*/AND 1=1`，利用注释干扰WAF解析。

## 三、防御与绕过对抗演进

| 防御措施         | 绕过技术           | 核心原理                       |
| ----------------| ----------------- | ---------------------------  |
| **黑名单过滤**   | 大小写/编码/注释符   | 黑名单无法覆盖所有变形           |
| **参数化查询**   | 无直接绕过方法       | 分离数据与指令，**最有效防御**   |
| **最小权限原则**  | 权限提升攻击        | 限制数据库账户权限              |
| **WAF规则**      | 多重编码/语义等价    | WAF难以解析复杂变形            |

> **关键结论**：正则清洗等黑名单机制对现代SQL注入基本无效（），因攻击者可使用宽字节、编码混淆、函数嵌套等技术绕过。**唯一可靠防御是参数化查询+最小权限+白名单校验**。例如Java中使用`PreparedStatement`绑定参数，而非拼接字符串；PHP中用`mysqli_real_escape_string`转义输入，但需注意此方法仍可能被宽字节注入绕过。
