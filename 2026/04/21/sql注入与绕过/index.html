sql注入与绕过

发表于2026-04-21|更新于2026-04-21
|浏览量:

SQL注入是通过将恶意SQL代码插入Web表单、URL参数或查询字符串中，利用应用程序对用户输入验证不足的漏洞，欺骗数据库执行非授权操作的安全威胁，其核心在于数据与指令的混淆，而绕过技术则针对安全防护机制设计，旨在规避检测规则。

一、SQL注入原理详解

1. 根本成因

  • 数据与指令混淆：应用程序将用户输入直接拼接到SQL查询中，未区分数据与指令的语义。
  • 过滤不严：对特殊字符（如单引号、分号）和SQL关键字（如SELECT、UNION）未进行有效过滤或转义。
  • 动态拼接：使用字符串拼接方式构造SQL语句，而非参数化查询。

3. 注入类型

  • 数字型注入：参数无引号包裹（如id=1），直接拼接条件（id=1 AND 1=2）。
  • 字符型注入：参数被引号包裹（如name='admin'），需闭合引号（name='admin'--）。
  • 盲注：无直接数据回显，通过布尔响应（真/假）或时间延迟（SLEEP(3)）推断数据。
  • 报错注入：利用数据库错误信息泄露数据（如AND (SELECT 1 FROM (SELECT COUNT(*),CONCAT(0x3a,version(),0x3a) AS x FROM information_schema.tables GROUP BY x))）。

二、常见绕过技术与原理

1. 编码绕过

  • URL编码：将单引号'编码为%27，空格编码为%20，如id=1%27%20AND%2011。
  • 双重URL编码：对已编码字符再次编码（%27→%2527），需服务端二次解码才能生效。
  • Unicode编码：用u0027替代单引号，部分WAF无法识别。
  • 十六进制编码：将字符串转换为十六进制（'admin'→0x61646D696E）。

2. 关键字混淆

  • 大小写混合：利用SQL不区分大小写特性，如SeLeCt、UnIoN。
  • 双写关键字：若过滤SELECT，可构造SELSELECTECT，过滤后剩余SELECT。
  • 注释符拆分：用/**/替代空格，如UNI/**/ON SEL/**/ECT 1,2。
  • 内联注释：MySQL支持/*!50000UNION*/语法，仅在特定版本执行。

3. 逻辑与函数替换

  • 逻辑符号替换：AND→&&，OR→||，如id=1%27||11。
  • 函数等价替换：SLEEP(3)→BENCHMARK(1000000,1)，SUBSTRING→MID。
  • 无逗号技巧：SUBSTR(database() FROM 1 FOR 1)替代SUBSTR(database(),1,1)。

4. 特殊场景绕过

  • 宽字节注入：GBK编码中%df%27组合为運'，使单引号逃逸（id=1%df%27 AND 11）。
  • 二次注入：先将恶意数据存入数据库（如admin' #），后续查询时触发注入。
  • 空字节绕过：在过滤器与应用处理差异处，用%00截断检测（category=book%00';DROP TABLE test）。

5. WAF对抗技巧

  • 动态查询：将SQL转为ASCII十六进制字符串执行（DECLARE @q VARCHAR(200); SET @q=0x44526f50205461526c65; EXEC(@q)）。
  • PHP正则回溯绕过：构造超长字符串（union/*100万个a*/select）触发pcre.backtrack_limit限制。
  • 参数污染：id=/*a=1*/1/*b=2*/AND 1=1，利用注释干扰WAF解析。

三、防御与绕过对抗演进

防御措施 绕过技术 核心原理
黑名单过滤 大小写/编码/注释符 黑名单无法覆盖所有变形
参数化查询 无直接绕过方法 分离数据与指令，最有效防御
最小权限原则 权限提升攻击 限制数据库账户权限
WAF规则 多重编码/语义等价 WAF难以解析复杂变形

关键结论：正则清洗等黑名单机制对现代SQL注入基本无效（），因攻击者可使用宽字节、编码混淆、函数嵌套等技术绕过。唯一可靠防御是参数化查询+最小权限+白名单校验。例如Java中使用PreparedStatement绑定参数，而非拼接字符串；PHP中用mysqli_real_escape_string转义输入，但需注意此方法仍可能被宽字节注入绕过。

四、实战建议

  1. 开发者：

    • 坚决使用参数化查询（预编译语句），避免动态拼接SQL。
    • 对必须拼接的字段（如表名），实施严格白名单校验（in_array($input, $valid_fields)）。
    • 数据库连接账户仅授予最小必要权限（如查询权限，禁用DROP/DELETE）。

  2. 安全人员：

    • 使用SQLMap时添加--tamper脚本（如unionalltounion.py）测试绕过能力。
    • 检查WAF是否支持多层解码（URL/HTML/Base64解码后检测）。
    • 通过ORDER BY N--探测字段数，用UNION SELECT 1,2,3定位显示位。

  3. 绕过测试：

    • 基础探测：id=1' AND 1=1 --（验证注入点）。
    • 字段数探测：id=1 ORDER BY 4 --（报错则字段数为3）。
    • 数据提取：id=-1 UNION SELECT 1,database(),3 --（获取数据库名）。

随着AI技术发展，LLM（大语言模型）可能被用于生成更复杂的绕过payload，如通过语义等价但字面不同的表达式绕过检测（）。防御需结合多层防护：输入规范化→正则深度匹配→上下文感知解析→最小权限控制，形成纵深防御体系。
